AgriTech · IoT · AI Intelligence · 2022

When a Pond Fails,
You Have Hours.

PondLogs was rebuilt as an AI-powered command system for precision aquaculture — transforming manual logs and fragmented sensor data into role-based intelligence that prevents crop failures before they happen.

Lead UI Designer AgriTech · IoT · AI Precision Aquaculture Dec 2021 – May 2022

At 3am, shrimp farms don't fail slowly.

A sudden oxygen drop can erase months of investment in hours. PondLogs had IoT sensors collecting real-time data from every pond — but the software layer couldn't translate those signals into action. Farm managers were still running on spreadsheets, instinct, and paper logs. The data was there. The decisions weren't.

No real-time pond intelligence

Farmers physically walked ponds to collect readings. No digital record of conditions. A single missed oxygen reading — especially at night — could end an entire crop with no warning.

Cognitive overload for managers

Managers juggled 50+ ponds via spreadsheets. Critical O₂ alerts were buried in the same feed as routine status updates with no urgency hierarchy. Nothing escalated automatically.

Role blindness across the farm

Farmers, managers, and owners all saw the same flat data. No role-specific intelligence. No one knew which signals were theirs to act on — so most waited, and some waited too long.

Three decisions that shaped everything.

The initial brief was "better dashboards." The insight was: farm teams needed decisional clarity, not more data. Every design choice had to pass one test — does this help someone act faster?

Role-based intelligence, not dashboards

Three distinct views — Farmer (what's at risk in my ponds right now?), Manager (where are my problem ponds?), Owner (am I profitable?). Same data, three lenses. Each role gets only what they need to act.

Biological urgency as design principle

Oxygen drops aren't notifications — they're survival events. Redesigned the alert hierarchy so biological emergencies bypass normal notification patterns and demand immediate attention. Criticality drove visual hierarchy.

Prediction over logging

Shifted the platform from passive data recording to AI-driven early warning. The system now tells you what's about to happen — not just what already did. Farmers stop reacting to failures and start preventing them.

Everyone had data.
No one had confidence.

PondLogs didn't need
better charts. It needed
to become decisive.

One system. Every pond. Every role.

Pond Health Command Center

Real-time O₂, temperature, pH, and feeding status for every pond in one view. Color-coded health states surface risk instantly. Tap any pond for deep diagnostics and trend history.

AI Alert Intelligence

Predictive alerts surface oxygen drops, feeding anomalies, and equipment failures before they become critical. Every alert ranked by biological urgency — not arrival time. The most dangerous issue is always visible first.

Role-Based Command Views

Farmer gets ground-level alerts (act now). Manager sees pond-health overview (oversee at scale). Owner tracks farm performance and ROI. Separate entry points — one platform for the entire organization.

REAP CASE STUDY
AI NetOps · Multi-Site Monitoring
AI NetOps Platform · Multi-Site Network Monitoring · 2026

When the Network Fails,
Every Site Feels It.

REAP is an AI-native network operations platform for monitoring firewalls, devices, links, routing paths, and incidents across multiple locations. It gives teams one live view of network health — then uses AI to explain what changed, where the issue is, what is affected, and what action should happen next.

AI NetOps Firewall Monitoring Topology Intelligence Runbook Automation

Network teams don't fail from lack of data. They fail from lack of shared context.

Every firewall, router, switch, and monitoring tool can produce alerts. But during an incident, engineers still need to manually connect the dots: which device changed, which link failed, which route shifted, which site is impacted, and whether the issue is safe to remediate. REAP was designed as a new command center for that moment — where the network is live, the topology is visible, and AI helps engineers move from alert noise to operational confidence.

No single live view across locations

Network teams often monitor sites, firewalls, devices, and incidents through separate tools. During an outage, the first question is still basic: where is the problem happening?

Alert overload without causality

A single device or link failure can generate many alerts. Engineers need to know the root event, not just every downstream symptom flooding into their queue.

Diagnosis depends on senior engineers

The fastest path to resolution often lives in the head of the most experienced engineer. REAP needed to make that reasoning available to the whole team — not just those who've seen it before.

Three decisions that shaped everything.

REAP was not designed as another dashboard. It needed to compress the full incident workflow — visibility, diagnosis, reasoning, action, and auditability — into one operational surface.

Map-first operations, not table-first monitoring

The global map became the starting point because network incidents are location-sensitive. Teams need to see affected sites, unhealthy links, and regional impact before they dive into device-level detail.

AI reasoning must show its evidence

REAP's AI could not behave like a black box. Every recommendation needed context: what happened, why it happened, confidence level, affected devices, supporting evidence, and the exact signals used.

Remediation needed control, not magic

The product could help execute runbooks, but engineers still needed trust and accountability. Sensitive actions require approval, command steps are visible, and every change is logged.

Everyone had alerts.
No one had the full story.

another monitoring dashboard.

QARun CASE STUDY
DevOps · QA SaaS
DevOps · QA SaaS · 2023

From API Chaos
to Calm Confidence.

Reframing QARun from a tooling problem into a systems decision — a redesign that aligned the product to how testers actually reason about API validation.

Lead UI Designer B2B SaaS · QA Platform Strategy + UX + System redesign

QA professionals were configuring, not validating.

QARun had the technical power to handle complex API testing. But the interface demanded too much from the people using it. Testers re-entered data across every test, validated through raw logs, and stitched together disconnected steps by memory. The workflow demanded too much — and returned too little confidence.

Repetitive data entry across every test

Endpoint URLs, auth headers, and payloads were re-entered manually for every test case. No reuse, no library, no connection between related tests. Every test started from scratch.

Validation through logs only

Test results came back as raw JSON and status codes. No visual confirmation layer. Every result required manual log parsing — a slow, error-prone process that killed throughput and confidence.

No connected test journeys

Tests existed as isolated API calls. There was no way to model a user journey — login → action → validation → logout. Real-world API behaviour was untestable because the tool didn't think in flows.

Five mental models. One legible product.

The platform was redesigned around how QA testers actually think — not how APIs technically work. Three principles drove every decision from information architecture to interaction design.

Reuse before recreate

Treat recurring test data as a reusable asset, not a repeated task. A centralised Data Library lets testers define variables once and reference them everywhere — endpoints, headers, auth tokens, payloads.

Flows over fragments

Design for test journeys and sequences, not disconnected API calls. Testers model real user behaviour — conditional sequences, chained assertions, shared session state — as a coherent flow, not a list of endpoints.

Confidence over configuration

Help users decide faster instead of drowning them in setup detail. Response previews, visual status indicators, and plain-language summaries replace raw JSON parsing as the primary validation surface.

QA testers think in flows,
not endpoints.

The platform had to match
how testers reason —
not how APIs work.

One unified test → execution → reporting loop.
